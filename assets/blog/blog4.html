Real-time voice communication is one of those features that transforms a multiplayer game from a collection of nameless avatars into a living, breathing social experience. Whether you're calling out enemy positions in a tactical shooter or coordinating a dungeon raid, voice chat creates the kind of spontaneous human connection that text chat simply can't replicate.

In this guide, I'll walk through my complete journey integrating Photon Voice 2 into an existing PUN2-based Unity game — from initial setup all the way to spatial audio and push-to-talk. I'll also share the gotchas I ran into (including a frustrating component-linking issue) and exactly how I solved them.

01 Why Photon Voice 2?

When evaluating voice solutions for Unity multiplayer games, Photon Voice 2 stands out for one key reason: it's built to work natively alongside PUN2. Both SDKs share the same Photon infrastructure, meaning:

  • Your players don't connect to a separate voice server — voice traffic travels through the same Photon relay network
  • Room management is synchronized — when a player joins or leaves a PUN room, the voice session follows automatically
  • The App ID setup is unified — you can use the same Photon App ID for both PUN and Voice
  • Latency is minimized through co-located data centers

Alternatives like Vivox or Agora are powerful but add third-party dependencies and require separate account management. For games already built on PUN2, Photon Voice 2 is the natural, friction-free choice.

02 Architecture Overview

Before diving into code, it helps to understand how Photon Voice 2 fits into the overall networking architecture. There are two parallel but linked connection layers:

PUN2 Layer — Game Data

Handles player positions, game state, RPCs, and all gameplay-related network messages. This is the layer you've already built your game on.

Voice Layer — Audio Data

Handles microphone capture, audio encoding via the Opus codec, transmission, and playback. This layer runs alongside PUN2 and is managed by a VoiceConnection component.

The bridge between these two layers is the PunVoiceClient component — a clever piece of Photon's SDK that listens to PUN's room events and mirrors them in the voice layer. It's what makes the two systems feel like one unified system.

03 Step-by-Step Integration

1. Import and Configure

After importing Photon Voice 2 from the Asset Store, open your PhotonServerSettings asset. You'll see a new Voice AppId field. You can either reuse your existing PUN AppId (simplest for small projects) or create a separate Voice App in the Photon Dashboard for independent analytics and scaling.

2. Set Up the Voice Connection

Add both VoiceConnection and PunVoiceClient to a persistent scene object — your NetworkManager works perfectly for this:

NetworkManager setup 
// On your NetworkManager (or dedicated AudioManager) GameObject:
// - Add Component: VoiceConnection
// - Add Component: PunVoiceClient
//
// PunVoiceClient automatically:
//   ✓ Connects to Voice when PUN connects
//   ✓ Joins a Voice room when PUN room is joined
//   ✓ Cleans up on disconnect — zero boilerplate
💡
Pro Tip Make the AudioManager object DontDestroyOnLoad so voice persists across scene transitions without reconnecting.

3. Player Prefab — Recorder + Speaker

The core voice components live on your Player Prefab. One Recorder captures the local player's microphone, while a Speaker on every player object plays incoming audio from that player.

PlayerVoiceSetup.cs 
public class PlayerVoiceSetup : MonoBehaviour
{
    [SerializeField] private Recorder recorder;

    void Start()
    {
        // Only activate recorder for the local player
        if (!photonView.IsMine) { recorder.enabled = false; return; }

        recorder.TransmitEnabled          = true;
        recorder.VoiceDetection           = true;   // Enable VAD
        recorder.VoiceDetectionThreshold  = 0.01f;
        recorder.SamplingRate             = SamplingRate.Sampling24000;
        recorder.FrameDuration            = OpusFrameDuration.Frame20ms;
    }
}

4. PhotonVoiceView — The Critical Link

PhotonVoiceView is the component that ties everything together. It maps a Photon Voice actor to a specific player object across the network. Without it, remote players won't hear audio from the correct source.

Your player prefab hierarchy should look exactly like this:

PlayerPrefab (root)
├── PhotonView
├── PhotonVoiceView ← links voice to the right player
├── Recorder ← assign in PhotonVoiceView inspector
├── Speaker ← assign in PhotonVoiceView inspector
└── AudioSource ← required by Speaker component

04 The Inspector Drag-and-Drop Problem

This is the issue that tripped me up — and likely trips up many developers. When you try to drag the Speaker component into the Speaker field of PhotonVoiceView in the Inspector, it simply doesn't work. The field seems to reject the assignment or stays empty.

The root cause is usually one of three things:

  • The Speaker is on a child GameObject, not the root — PhotonVoiceView only looks at the same transform
  • The field is greyed out because PhotonVoiceView has auto-linking enabled and manages the reference internally
  • The AudioSource component is missing — Speaker requires it on the exact same object

Fix 1: Assign at Runtime via Code (Most Reliable)

Skip Inspector assignment entirely and wire everything up in Awake():

VoiceSetup.cs — Awake() 
void Awake()
{
    var pvv    = GetComponent<PhotonVoiceView>();
    var rec    = GetComponent<Recorder>();
    var spk    = GetComponent<Speaker>();

    pvv.RecorderInUse = rec;
    pvv.SpeakerInUse  = spk;
}

// Verify at runtime
void Start()
{
    var pvv = GetComponent<PhotonVoiceView>();
    Debug.Log("Recorder linked: " + (pvv.RecorderInUse != null));
    Debug.Log("Speaker linked:  " + (pvv.SpeakerInUse  != null));
}
✅
Zero-Code Alternative Leave both fields completely blank in the Inspector. If Recorder, Speaker, and PhotonVoiceView are all on the same root GameObject, auto-detection handles linking at runtime with no extra code.

05 Advanced Features

Push-to-Talk

For competitive games where ambient noise is a concern, push-to-talk keeps voice channels clean. Just toggle TransmitEnabled based on key input:

PushToTalk.cs 
void Start()  => recorder.TransmitEnabled = false; // Start muted

void Update()
{
    if (!photonView.IsMine) return;
    recorder.TransmitEnabled = Input.GetKey(KeyCode.V);
}

Spatial / Proximity Audio

For open-world or large map games, spatial audio adds enormous immersion. Players hear each other only when nearby, with volume that falls off naturally with distance. This is achieved through Unity's AudioSource 3D settings — since Speaker plays through the AudioSource on the remote player's GameObject, spatial positioning is automatic:

ProximityVoice.cs 
void Start()
{
    if (photonView.IsMine) return; // Only configure remote players

    voiceAudioSource.spatialBlend  = 1.0f;               // Full 3D
    voiceAudioSource.maxDistance   = 20f;
    voiceAudioSource.rolloffMode   = AudioRolloffMode.Linear;
    voiceAudioSource.dopplerLevel  = 0f;                  // Disable doppler
}

Mute / Volume UI Controls

VoiceUIControls.cs 
public void OnMuteToggled(bool isMuted)
{
    localRecorder.TransmitEnabled = !isMuted;
}

public void OnVolumeChanged(float value)
{
    AudioListener.volume = value; // Global or target specific AudioSource
}

06 Common Pitfalls & Checklist

⚠️
PitfallRecorder active on remote players — always guard with photonView.IsMine before enabling Recorder. Skipping this causes every player to capture all microphones simultaneously.
⚠️
PitfallMissing AudioSource — Speaker requires an AudioSource on the same GameObject to function. This is the silent killer of many setups.
⚠️
PitfallWrong App ID — Voice needs its AppId configured under the Voice application in the Photon Dashboard, not just the Realtime/PUN app.
⚠️
PitfallNo PunVoiceClient — without this, the voice layer never joins the room automatically. Players will connect to PUN but voice remains silent.

Here's the final component checklist for your player prefab:

🔷
PhotonViewCore PUN identity — required on all networked objects
🎙️
RecorderLocal player only — capture mic input. Disable for remotes.
🔗
PhotonVoiceViewLinks voice actor to PUN player. The critical connector.
🔊
Speaker + AudioSourceRemote players only — plays incoming voice audio.

07 Conclusion

Integrating Photon Voice 2 with PUN2 is well worth the investment. Once the component hierarchy clicks — VoiceConnection and PunVoiceClient on the manager, Recorder, Speaker, AudioSource, and PhotonVoiceView on the player prefab — the system is remarkably solid and low-maintenance.

The key insight that unlocked everything for me was understanding that PhotonVoiceView is the glue between the two systems, and that if Inspector assignment fails, code-based wiring in Awake() is both simpler and more reliable.

From here, you can layer in push-to-talk, team-based channels, spatial audio, and mute controls — all building on the same foundation. Happy building!
