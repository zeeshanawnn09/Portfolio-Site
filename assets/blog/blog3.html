01 Synchronization Requirements

In multiplayer games, synchronization is everything. When assigning secret roles like Ghost and Priest, you must ensure:

  • All players receive the same result
  • The assignment happens only once
  • Late joiners stay in sync
  • The system is secure from duplication

In this post, we'll break down an RPC-centric implementation using Photon Unity Networking (PUN 2) and explain:

  • Why RPCs were used
  • How they were implemented
  • What benefits they provide
  • When this approach is ideal

02 Why Go RPC-Centric?

Photon offers multiple synchronization methods, each suited to different use cases:

Method Good For
Custom Properties Persistent data
RaiseEvent Lightweight event broadcasting
RPC Direct object-bound execution

In this system, RPCs were chosen because:

  • Roles must trigger behavior on a specific player object
  • Assignment should execute immediately
  • Late joiners must receive the role
  • Logic stays tightly coupled (role → character behavior)

03 Master Client Authority

Only the Master Client assigns roles. If every client shuffled locally, you would get desynchronization issues. Master authority ensures:

Authority Check 
if (!PhotonNetwork.IsMasterClient)
    return;
  • Only one shuffle happens
  • Only one role list exists
  • Everyone receives identical results

04 Role Pool Creation + Shuffle

First, build the role pool based on the required counts:

RolePoolSetup.cs 
List<PlayerRole> roles = new List<PlayerRole>();

for (int i = 0; i < PRIEST_COUNT; i++)
{
    roles.Add(PlayerRole.Priest);
}
roles.Add(PlayerRole.Ghost);

Then apply a Fisher-Yates shuffle to randomize role distribution:

FisherYatesShuffle.cs 
for (int i = roles.Count - 1; i > 0; i--)
{
    int randomIndex = Random.Range(0, i + 1);

    PlayerRole temp = roles[i];
    roles[i] = roles[randomIndex];
    roles[randomIndex] = temp;
}

This approach is scalable and prevents duplicate role assignment.

05 RPC Role Distribution

Once roles are shuffled, the Master Client sends each role to its target player via RPC:

RoleAssignment.cs 
pControl.photonView.RPC(
    "AssignPlayerRole",
    RpcTarget.AllBuffered,
    (int)role
);

Why RpcTarget.AllBuffered?

  • All → Executes on every connected client
  • Buffered → Stores call for late joiners
💡
Late Join Protection Without buffering, reconnecting players would not receive a role. Buffered RPCs replay automatically when new players join the room.

06 Player Role Execution

Each player object receives the RPC and executes role-specific logic locally:

PlayerControls.cs 
[PunRPC]
void AssignPlayerRole(int roleIndex)
{
    PlayerRole role = (PlayerRole)roleIndex;

    if (role == PlayerRole.Ghost)
    {
        BecomeGhost();
    }
    else
    {
        BecomePriest();
    }
}

By attaching the RPC to the player object, the role logic stays clean and modular. Each player controls its own behavior based on the assigned role.

07 Benefits of an RPC-Centric Architecture

1. Strong Authority Model

Only the Master assigns roles, preventing duplication and ensuring consistent state across all clients.

2. Automatic Late Join Sync

Buffered RPCs replay automatically when new players join, eliminating manual sync logic.

3. Clean Separation of Concerns

  • RoomController → Manages game flow and orchestration
  • PlayerControls → Handles individual character behavior

4. Event-Driven Execution

No need for polling or update checks. The event executes instantly when triggered.

5. Scalable Design

Want multiple ghosts? Special roles? Just change role list creation. The RPC distribution logic stays identical.

08 Multiplayer Safety Considerations

⚠️
Prevent Double Assignment Always guard your role assignment logic with a flag to prevent duplicate shuffles and inconsistent state.
SafetyCheck.cs 
private bool rolesAssigned = false;

void AssignRoles()
{
    if (rolesAssigned) return;
    rolesAssigned = true;
    
    // Assignment logic here...
}

Handle Master Client Switch

If the Master Client disconnects, Photon automatically assigns a new Master. You may want to re-validate state:

MasterClientHandler.cs 
public override void OnMasterClientSwitched(Player newMasterClient)
{
    // Optional: Re-validate state
    // Check if roles were already assigned
}

09 Final Multiplayer Flow

The complete role assignment sequence follows this order:

  1. Master loads room
  2. Players join sequentially
  3. Characters spawn in the game world
  4. When 4 players present:
    • Create role list
    • Shuffle roles using Fisher-Yates
    • Send buffered RPCs to each player
  5. Players execute role logic locally
  6. Late joiners receive buffered calls automatically

10 When Should You Use RPC-Centric Design?

Use RPC when:

  • The action is event-based (one-time triggers)
  • Immediate execution is required
  • Logic belongs to a specific object
  • Late join replay is necessary

Avoid RPC when:

  • Persistent long-term data is required (use Custom Properties instead)
  • Server-side validation is needed
  • High-frequency event spam occurs (60+ calls per second)

11 Final Thoughts

This RPC-centric implementation provides a clean, scalable, and multiplayer-safe architecture built around one simple principle:

One authority decides.
RPC distributes.
Objects react locally.

That's solid multiplayer engineering. The pattern scales beautifully — whether you're assigning roles in a social deduction game or distributing loot in a dungeon crawler, the core principle remains the same: centralized decision-making with distributed execution.
